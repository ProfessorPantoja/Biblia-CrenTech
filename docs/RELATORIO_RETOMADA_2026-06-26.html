🔧Achados Técnicos (por severidade)

crítico resolver antes de divulgar alto importante p/ produção médio qualidade/manutenção baixo polimento

crítico Chave da API do Gemini exposta no navegador

Em vite.config.ts a GEMINI_API_KEY é injetada direto no bundle do cliente (process.env.API_KEY). Isso significa que qualquer pessoa que abrir o app consegue extrair sua chave do JavaScript e usá-la — gerando custo na sua conta Google e risco de abuso.

Como resolver: mover as chamadas do Gemini para um pequeno backend/serverless (ex.: Vercel Functions, já que o deploy é Vercel) que guarda a chave no servidor. Alternativa rápida: restringir a chave por domínio/HTTP referrer e cota no Google Cloud Console enquanto não há backend.

crítico Modelo default gemini-3-flash-preview que não existe no ambiente

Em services/geminiService.ts o modelo é import.meta.env.VITE_GEMINI_MODEL || 'gemini-3-flash-preview', mas não existe VITE_GEMINI_MODEL no .env.local. Ou seja, hoje o app tenta sempre um modelo "preview" como padrão. Modelo preview como default de produção é arriscado (pode sair do ar/mudar nome) e, se falhar, gasta uma chamada extra antes de cair no fallback gemini-2.5-flash.

Como resolver: definir um modelo estável e atual como default (ex.: gemini-2.5-flash) e só usar preview quando explicitamente setado no .env. Validar qual é o modelo Gemini mais recente disponível na sua conta antes de fixar.

alto Tailwind via CDN (cdn.tailwindcss.com)

O index.html carrega o Tailwind por um <script> de CDN. O próprio Tailwind avisa que isso não é para produção: depende de internet, é mais lento, não faz "tree-shaking" (manda CSS demais) e pode piscar sem estilo no carregamento.

Como resolver: instalar o Tailwind como dependência de build (PostCSS) e gerar o CSS no vite build. De quebra resolve o item do index.css abaixo.

alto PWA não funciona offline de verdade (sem Service Worker)

Existe manifest.json e a Bíblia está em 66 JSONs locais, mas não há Service Worker. Então o app não abre offline e os JSONs não ficam em cache — a promessa de "Bíblia offline" do CHANGELOG não se cumpre na prática.

Como resolver: adicionar vite-plugin-pwa (workbox) para cache de assets e dos JSONs bíblicos. É um ganho enorme de percepção de qualidade com pouco esforço.

alto index.css referenciado mas inexistente (404)

O index.html tem <link rel="stylesheet" href="/index.css">, mas o arquivo não existe. O build avisa isso explicitamente. Hoje não quebra nada só porque o Tailwind CDN cobre os estilos, mas é um 404 em toda visita.

Como resolver: criar o index.css (e usá-lo como entrada do Tailwind no build) ou remover o link. Resolve junto com a migração do Tailwind.

médio 5 erros de type-check (import.meta.env sem tipagem)

npx tsc --noEmit retorna 5 erros do tipo "Property 'env' does not exist on type 'ImportMeta'" em geminiService.ts, useWakeLock.ts e supabaseClient.ts. O build passa (Vite usa esbuild, que ignora tipos), mas o projeto não está "type-clean".

Como resolver: adicionar "vite/client" ao types do tsconfig.json (ou criar um vite-env.d.ts). Correção de 1 linha.

médio Dois sistemas de histórico divergentes

O AppContext salva o histórico dentro da chave bible_crentech_data. Mas existe um services/HistoryService.ts usando outra chave (bible_crentech_history) que não é importado em lugar nenhum (órfão). Quando você for fazer a tela de Histórico, é fácil cair na fonte errada.

Como resolver: decidir uma única fonte de verdade (recomendo a do AppContext) e apagar ou reescrever o HistoryService em cima dela.

médio Código morto: supabaseClient.ts e pixGenerator.ts

Ambos existem mas nenhum outro arquivo os importa. O Supabase está conectado (chaves no .env) porém ocioso. Isso aumenta o bundle e confunde quem lê o código.

Como resolver: remover por enquanto (dá pra recuperar do git quando for usar), OU manter só se houver plano concreto de usar Supabase já na retomada.

médio Padrões de variáveis de ambiente misturados

Convivem process.env.API_KEY (injetado via define) e import.meta.env.VITE_*. Dois jeitos de fazer a mesma coisa dificultam manutenção e foram parte da causa dos erros de tipo.

Como resolver: padronizar tudo em import.meta.env.VITE_* (padrão nativo do Vite).

baixo Bundle único de 523 kB sem code-splitting

O Vite avisa que o chunk passou de 500 kB. Não é grave (132 kB gzip), mas dá pra melhorar o tempo de primeira carga.

Como resolver: import() dinâmico nas telas mais pesadas (Reader, Search) e/ou manualChunks.

baixo Acessibilidade: zoom bloqueado

O viewport usa maximum-scale=1.0, user-scalable=no, impedindo o usuário de dar zoom — ruim para acessibilidade, especialmente num app de leitura usado por pessoas mais velhas.

Como resolver: remover user-scalable=no e maximum-scale.

baixo 26 console.* espalhados + sem lint/CI/testes

São 26 chamadas de console que vão para produção. Não há ESLint/Prettier configurado, nem testes (o test_crc.cjs na raiz é um script solto). Para um projeto que você quer manter, vale uma base mínima de qualidade.

Como resolver: ESLint + Prettier, limpar logs de debug, e (opcional) um teste de fumaça em useBible/parser de referências.
