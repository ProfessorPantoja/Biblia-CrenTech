🔑 Cheat Sheet — Login com Google (OAuth)

Receita reutilizável pra qualquer app. Serve com Supabase, Firebase, Auth0 ou backend próprio.
O fluxo é sempre o mesmo: Google emite a identidade → seu "porteiro" (provedor/backend) recebe e cria a sessão.
💡 A sacada do limite de projetos: você não precisa de 1 projeto Google Cloud por app. Um único projeto pode ter vários Client IDs (um por aplicação). Só crie projetos separados quando quiser uma tela de consentimento com nome/marca diferente por app. Assim o limite (~12 projetos) raramente incomoda.

📚As peças (decore isto e o resto flui)

Projeto (Google Cloud)
O "guarda-chuva". Contém a tela de consentimento e os clientes OAuth.
Tela de consentimento
O que o usuário vê ("App X quer acessar sua conta"). É por projeto.
Client ID
Identificador público do app. Pode aparecer no front-end, sem problema.
Client Secret
Senha secreta do app. NUNCA no front-end/git. Só no servidor/provedor.
JavaScript Origins
De quais domínios o navegador pode iniciar o login (ex: http://localhost:3000).
Redirect URI
Pra onde o Google devolve o usuário após autenticar. Tem que bater exatamente.
Scopes
O que você pede (email, perfil...). Pra login simples, o básico já vem.
Test users
Enquanto o app está "em teste", só e-mails nesta lista conseguem logar.

🧭A receita (sempre os mesmos passos)

no Google Cloud

1
Projeto: crie um novo ou reutilize um existente (lembre da sacada acima).
2
Tela de consentimento: tipo Externo, preencha nome do app + e-mails. Adicione seu e-mail em Test users.
3
Credenciais → ID do cliente OAuth → Aplicativo da Web.
4
JavaScript Origins: os domínios do app (dev + produção). Sem caminho, sem barra final.
http://localhost:3000   https://meuapp.com
5
Redirect URIs: o endereço de callback (veja a regra de ouro abaixo). Tem que ser exato.
6
Criar → copie Client ID e Client Secret.

no seu provedor (Supabase/Firebase/Auth0/backend)

7
Ative o provedor Google e cole o Client ID + Secret. Configure as URLs de retorno permitidas (Site URL + Redirect URLs). Salve.

⭐Regra de ouro do Redirect URI

É o que mais quebra. O Redirect URI no Google tem que ser o callback de quem recebe o login — e isso muda conforme a arquitetura:

Você usa...Redirect URI no Google é...
Supabasehttps://SEU-PROJETO.supabase.co/auth/v1/callback
Firebasehttps://SEU-APP.firebaseapp.com/__/auth/handler
Auth0https://SEU-TENANT.auth0.com/login/callback
Backend própriohttps://seuapp.com/auth/google/callback (a rota que VOCÊ criar)
⚠️ Erros clássicos: barra / a mais no final, http em vez de https, ou usar o domínio do app quando deveria ser o do provedor. 1 caractere errado = login falha.

🚦Modo Teste vs Publicado

  • Em teste (Testing): só os e-mails da lista Test users conseguem logar. Aparece um aviso "app não verificado" → é normal, clica em "Avançar/Continuar".
  • Publicado (In production): qualquer pessoa loga. Pra liberar geral, vá na tela de consentimento e clique em "Publicar app".
  • Verificação do Google: só é exigida se você pedir scopes "sensíveis" (Gmail, Drive, etc). Pra login básico (email/perfil), não precisa de verificação.

🩺Tabela de erros (decora que vira diagnóstico instantâneo)

ErroSignificaConserto
redirect_uri_mismatchO Redirect URI não bate com o cadastrado.Compare caractere a caractere no Google (https, sem barra final, domínio do provedor).

🔒Segurança (a regra que nunca muda)

  • Client Secret = só no servidor/provedor. Nunca no front-end, nunca no git.
  • Se um secret vazar (ex: apareceu num print), revogue e gere outro — leva segundos.
  • Chaves "públicas" (Client ID, anon/publishable key) podem ficar no front — a proteção real é o RLS / regras do provedor.
  • Em produção, cadastre as variáveis no painel do deploy (ex: Vercel → Environment Variables), não no código.
Resumo de bolso: 1) projeto → 2) consent screen + test users → 3) client web → 4) origins (domínios) → 5) redirect URI (callback do provedor) → 6) copia ID+Secret → 7) cola no provedor e ativa. Erro? Olha a tabela. 🚀
